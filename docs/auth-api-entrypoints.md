# API Auth Entrypoints and Contributor Extension Rules

This document defines the current auth entrypoints and the safe way to extend them.

## Entrypoints
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/logout`
- `GET /api/v1/auth/me`

## Contracts
- Request/response payloads must use `@lumen/types` (`packages/types/src/auth.ts`).
- Route handlers in `apps/api/src/auth/router.ts` should keep explicit `LoginRequest`, `LoginResponse`, `RegisterRequest`, and `RegisterResponse` typing.

## Token Signing Extension Rule
- Access token signing is abstracted in `apps/api/src/auth/token-signer.ts`.
- Contributors must depend on `AccessTokenSigner` instead of inlining token generation.
- Rotation strategy: replace `StaticKeyAccessTokenSigner` with key-ring backed signer while keeping the same interface.

## Contributor Guardrails
- Keep auth response payloads backward compatible unless milestone notes require changes.
- Keep error responses normalized via `normalizeAuthError`.
- Add verification notes/tests when auth contract behavior changes.

## Verification Notes
- Register and login must return typed `session.accessToken` values generated by the signer abstraction.
- `GET /auth/me` behavior remains unchanged in this slice.
- Web auth screens should use `NEXT_PUBLIC_API_BASE_URL` and surface loading, error, and retry states for both login and registration submissions.
