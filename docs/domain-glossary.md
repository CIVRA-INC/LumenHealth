# Domain Glossary

> Version: 1.0.0 — CHORD-014
> Owner: Architecture & Standards
> Review trigger: new domain entity, renamed concept, or cross-team terminology conflict

---

## Purpose

This glossary is the single source of truth for domain terminology used across product, engineering, and blockchain layers of LumenHealth. All code identifiers, API field names, database collection names, and documentation should use the terms defined here. When a term is ambiguous or overloaded, this document takes precedence.

---

## Core Clinical Terms

### Clinic

An organization registered on the LumenHealth platform. A clinic owns all patient records, encounters, staff accounts, and billing configuration within its boundary. Every multi-tenant data record carries a `clinicId` foreign key.

- **Code identifier:** `clinic`, `clinicId`
- **Collection:** `clinics`
- **Owner:** `apps/api/src/modules/auth` (registration), `apps/api/src/modules/patients` (scoping)

### Patient

An individual whose health information is managed by a clinic. A patient record contains demographics and a timeline of encounters. Patients are scoped to a single clinic.

- **Code identifier:** `patient`, `patientId`
- **Collection:** `patients`
- **Owner:** `apps/api/src/modules/patients`

### Encounter

A single clinical visit or interaction between a patient and a care provider. An encounter is append-only — once created it can be extended with vitals, notes, and diagnoses, but its core record is never deleted. An encounter moves through a lifecycle: `open → claimed → closed`.

- **Code identifier:** `encounter`, `encounterId`
- **Collection:** `encounters`
- **Owner:** `apps/api/src/modules/encounters`
- **States:** `open`, `claimed`, `closed`

### Vital

A discrete physiological measurement recorded during an encounter (e.g., blood pressure, temperature, weight). Vitals are always associated with an encounter and a patient.

- **Code identifier:** `vital`, `vitalId`
- **Collection:** `vitals`
- **Owner:** `apps/api/src/modules/encounters` (sub-resource)

### Note

A free-text clinical observation or SOAP note attached to an encounter. Notes are authored by a staff member and are immutable after submission.

- **Code identifier:** `note`, `noteId`
- **Collection:** `notes`
- **Owner:** `apps/api/src/modules/encounters` (sub-resource)

### Diagnosis

A coded clinical finding attached to an encounter, referencing an ICD-10 code. A single encounter may have multiple diagnoses.

- **Code identifier:** `diagnosis`, `diagnosisId`
- **Collection:** `diagnoses`
- **Owner:** `apps/api/src/modules/encounters` (sub-resource)

### Queue

The real-time list of open encounters waiting to be claimed by a care provider. The queue is clinic-scoped and updated via server-sent events.

- **Code identifier:** `queue`, `queueEntry`
- **Owner:** `apps/api/src/modules/encounters` (queue sub-resource)

## Staff and Identity Terms

### User

A staff member with an account on the LumenHealth platform. A user belongs to exactly one clinic and carries a role that governs their permissions.

- **Code identifier:** `user`, `userId`
- **Collection:** `users`
- **Owner:** `apps/api/src/modules/auth`

### Role

A named permission set assigned to a user. Current roles: `admin`, `doctor`, `nurse`, `receptionist`.

- **Code identifier:** `role`
- **Owner:** `apps/api/src/modules/auth`

### Session

An authenticated HTTP session represented by a JWT access token and a refresh token. Sessions are stateless on the server; the refresh token is stored for revocation.

- **Code identifier:** `session`, `accessToken`, `refreshToken`
- **Owner:** `apps/api/src/modules/auth`

## Payment and Blockchain Terms

### Payment Intent

A pending payment request generated by the API before a Stellar transaction is initiated. A payment intent records the expected amount, currency, and memo, and transitions to `confirmed` or `expired`.

- **Code identifier:** `paymentIntent`, `paymentIntentId`
- **Collection:** `paymentIntents`
- **Owner:** `apps/api/src/modules/payments`
- **States:** `pending`, `confirmed`, `expired`

### Stellar Transaction

An on-chain payment executed on the Stellar network. The transaction hash is the canonical proof of payment. Stellar operations are isolated to `apps/stellar-service`.

- **Code identifier:** `stellarTxHash`, `transactionHash`
- **Owner:** `apps/stellar-service`

### Subscription

A recurring billing arrangement that grants a clinic access to the platform. Subscription status gates certain API features.

- **Code identifier:** `subscription`, `subscriptionStatus`
- **Collection:** `subscriptions`
- **Owner:** `apps/api/src/modules/payments`
- **States:** `active`, `trialing`, `past_due`, `canceled`

## AI Terms

### Visit Summary

An AI-generated narrative summary of an encounter, produced by the Gemini integration. A visit summary is advisory — it does not replace clinician documentation.

- **Code identifier:** `visitSummary`
- **Owner:** `apps/api/src/modules/ai`

### CDS (Clinical Decision Support)

Background AI analysis that surfaces key clinical insights during an encounter. CDS output is streamed to the frontend and is not persisted.

- **Code identifier:** `cds`, `cdsWorker`
- **Owner:** `apps/api/src/modules/ai`

## Infrastructure Terms

### Monorepo

The single Git repository containing all LumenHealth apps and packages, managed by Turborepo and NPM Workspaces.

### Workspace

An individual app or package within the monorepo (`apps/api`, `apps/web`, `apps/stellar-service`, `packages/config`, `packages/types`).

### Shared Contract

A TypeScript interface, Zod schema, or API response shape defined in `packages/types` and consumed by more than one workspace. Changes to shared contracts must follow `CONTRACT_VERSIONING.md`.

### Audit Log

An immutable record of a mutating action performed by a user. Audit logs capture the actor, clinic, resource, action, and timestamp. They are append-only and must never be deleted.

- **Code identifier:** `auditLog`, `auditLogId`
- **Collection:** `auditLogs`
- **Owner:** `apps/api/src/modules/audit-logs`

## Naming Conventions

| Context | Convention | Example |
|---|---|---|
| TypeScript identifiers | camelCase | `patientId`, `clinicId` |
| MongoDB collection names | camelCase plural | `patients`, `encounters` |
| API route segments | kebab-case | `/audit-logs`, `/payment-intent` |
| Environment variables | SCREAMING_SNAKE_CASE | `MONGO_URI`, `JWT_ACCESS_TOKEN_SECRET` |
| File names | kebab-case | `patient.service.ts`, `encounter.router.ts` |

## Deprecated Terms

| Deprecated | Preferred | Notes |
|---|---|---|
| `visit` | `encounter` | Use `encounter` in all new code |
| `client` | `patient` | Use `patient` in all new code |
| `provider` | `user` with role `doctor` | Role-qualified user is preferred |
